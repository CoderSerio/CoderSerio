分析

JavaScript主要是由两个部分组成:

ECMAScript标准
ECMA，欧洲计算机制造商协会（European Computer Manufacturers Association）的简称
规定了JavaScript语法的标准

W3C标准
W3C，万维网联盟(World Wide Web Consortium)的简称，
规定了WebAPI相关的标准，比如DOM和BOM

当然，两大官方标准在实际上可能并不被完全遵守，比如BOM的标准，
可能因为浏览器厂商不同而导致实际使用情况不同

JS中的内存模型

严格地说，JS并没有区分堆栈内存，但是我们依旧把它看做是两个部分——堆内存，栈内存，这样有助于我们理解JS对于不同数据类型的处理，以及如何进行内存管理

另外，不知道大家有没有想过，栈内存之所以叫做栈内存是因为可以把它看成一个栈，那么问题来了，堆内存和堆有什么关系呢？

答案是，现代意义上的堆内存和堆没有关系。

Unicode字符代码

JS字符串是Unicode字符代码，更准确的说是UTF-16字符编码

注意，字符代码和字符编码是两个概念

字符代码
特定字符在某个字符集中的序号

字符编码
在传输、存储过程当中用于表示字符的以字节为单位的二进制序列

具体地说，Unicode字符代码可以是每个字符在Unicode表里面的序号，而UTF是一种类似哈夫曼编码的过程

这里就简单提一下编码的发展过程:

  1. 开始是只有我们熟知的ASCII字符代码
  2. 然后各个国家得有编码兼容文字，比如我国的GB2312
  3. 后来出现Unicode编码，将大部分内容都字符成2个字节，然后形成Unicode字符代码
  4. 基于Unicode编码又出现了UTF-8编码，能根据数据大小动态编码成1-6个字节
    （英文通常1，中文通常3），然后形成Unicode字符代码
    在内存中的时候都是unicode，而在硬盘和文本编辑器中都是utf-8

NaN的判断

至于结果是否为NaN，可以通过两种方式判断

isNaN()和Number.isNaN()

前者是Window.isNaN()，它们的判断标准不太一样，前者并不是后者的语法糖

注意，不能用==或===判断NaN，就像是无穷大不等于无穷大一个道理

它们的区别在于：

Number.isNaN()
宽泛标准，会用尽一切办法对判断的数据进行类型转换直到数据变成Number类型，或者没有方法可用，才会得出判断结果

isNaN()
严格标准，直接判断而不会试图转换数据类型

这就意味着，如果Number.isNaN()判断的结果为true，那么传入的数据就绝对无法通过JS类型转换变成Number

console.log(Number.isNaN('123asdasdas')) 
// false
console.log(Number.parseInt('123asdasdas')) 
// 123，意味着可以转化

console.log(isNaN('123asdasdas')) 
// true

要转换数据类型之前，我们先要有判断数据类型的方法:

typeof
返回变量类型
用法类似于sizeof，或许也是运算符吧
能够准确判断null以外的基本数据类型，
遵循以下规则

  • null判断为object（历史遗留Bug）
  • 函数/方法判断为function
  • 其余引用类型全部判断为object

instanceof
返回一个布尔值
用法是：a instanceof A ，判断a是否为A的实例对象（或者说A是否为a的构造函数）
基本数据类型传入返回false

tip：
如果是以字面量赋值形式创建的基本数据类型变量，则没有构造函数；
以构造函数创建的基本数据类型变量，实质上是对象，是引用数据类型。

另外，这里附上instanceof的底层实现

function myInstanceof(a, A) {
    if (typeof a !== 'object' || a === null) 
        return false;
    let proto = Object.getPrototypeOf(a); // 获取a.__proto__
    while(true) {                  
        if(proto === null) 
            return false;
        if(proto === A.prototype)
            return true;        
        proto = Object.getPrototypeof(proto);
    }
}

代码看着有那么多，其实就是什么，就是遍历链表找有无目标值嘛
（当然，首先要判断一下是不是对象，不是对象那肯定没有构造函数的说）
找到null就是遍历完了都没找到目标值，就返回false；找到了自然就返回true

​ 这里说明一下Number.parseInt和parseInt的关系：

一开始只有parseInt，后来ES6规范来了，觉得你这不对啊，怎么一个裸的方法挂window上啊——然后parseInt就被扔到Number上 了。但是为了向后兼容，parseInt就像是种民俗习惯一样被保留下来了。

它们在效果上没有区别。

Number()转换

1.对于Boolean，true、false分别返回1、0
2.对于Number，返回原值
3.对于null，返回0
4.对于undefined，返回NaN
5.对于String：

  • 如果字符串只包含 数字以及正负号，就尽可能转为符合人类思维的正确的值。即：“1”返回1，“3413”返回3413，’’-33”返回-33，‘-011’返回-11
  • 如果字符串中包含 有效的小数格式，则转换为对应的小数，同样地会忽略掉前缀0
  • 有效的16/8/2进制格式，返回对应大小的十进制数值
  • 如果是 空串或者空格串，返回0
  • 非以上格式的字符串返回NaN

6.如果是Object:

  • 先处理其valueof方法的返回值，按照前面的规则返回值。如果返回值为NaN，则处理其toString()方法的返回值，把结果放入Number()进行转换，再按前面的规则返回值

parseInt()转换

1.对于String:

  • 忽略所有前置空格后，第一个字符如果是 数字 或 正负号，
    则继续向后读取，直到解析到非数字字符为止；
  • 忽略所有前置空格后， 第一个字符如果不是 数字或正负号，则返回NaN
    (空串也是NaN)

2.对于Number:

  • 若为16进制或10进制，则返回对应的10进制值
  • 可传入第二个参数，取值范围是[2,36]，表示第一个参数所采用的进制，然后返回其10进制值。
    这种情况下，第一个参数可以省略进制的特殊标识，比如’0x’,’0b’等
    比如 parseInt(‘ff’,16) 返回 256

parseFloat()转换

和parseInt一样，只是可以解析成浮点数。
parseFloat不支持第二个参数。

另外，这里再补充一下JS的逻辑大小判断规则

===三等号

和==的区别及联系:

第一步： == 比较 两端数据类型，类型不同则进入第二步，类型相同则进入第三步
第二步： 发生一次类型转换，进入第三步
第三步：===比较，若为基本数据类型，则依次比较数据类型、数据值，出现不同立刻终止返回false，全部相同 则返回true；若为引用数据类型，则依次比较数据类型、数据值、地址，过程同上。

典型例子

null == undefined // true, 都可以转为false和0
null === undefined // false

Object.is()
和三等号类似的，但是做了一定的调整，比如

这两种严格比较最主要的区别也就是对上述两个值的比较了，其余效果基本一致

Object.is()判断上述两者的关键代码如下：

// 实现判断NaN等于NaN

组合式继承

既然，原型链继承能继承原型上的，构造函数继承能继承实例上的——那么它们配合一下不就完美了吗

function Science (chineseName, stand) {
    this.chineseName = chineseName
    this.stand = stand
}
Science.prototype.showChineseName = function () {
    console.log(this.chineseName)
}
function Physics (chineseName, stand) {
    // 第一次调用父类构造函数
    Science.call(this, chineseName, stand)
}
// 第二次调用父类构造函数
Physics.prototype = new Science()

const subject = new Physics("物理", "Albert Einstein")
subject.showChineseName() // "物理"

缺点是会调用两次父类的构造函数

原型式继承

注意别和原型链继承弄混了！原型式继承的核心是把整个父级对象，作为子级的原型

function Base () {
    this.name = '基石'
}
Base.prototype.nickName = '石头蛋子'

function Stone (Parent) {
    function middleWare () {}
    middleWare.prototype = Parent
    return new middleWare()
}

// 注意这里可不是new
const stone = Stone (new Base())
console.log(stone.name, stone.nickName) // 基石 石头蛋子

原型式继承非常出色，它这样的设计能够直接获取到父级上的所有数据（包括原型链上的）

但是存在的问题是，它把临时的中间类middleWare直接实例化了，所以我们没办法在子级的原型链上添加数据

寄生式继承

寄生式继承就是为了解决原型式继承的问题——不能继续在原型连上添加东西

和原型式的代码非常相似，关键就在于它return的内容由一个对象变成了一个类

function Base () {
    this.name = "基石"
}
Base.prototype.nickName = '石头蛋子'

function Stone (obj) {
    function middleWare () {}
	middleWare.prototype = obj
	// 关键就在这里，没有new
    return middleWare
}

// 注意这里可不是new
const StoneClass = Stone(new Base())
StoneClass.prototype.intro = function () {
    console.log("我是" + this.nickName)
}
const stone = new StoneClass()
stone.intro() // 我是石头蛋子

寄生组合式继承

寄生组合实际上是寄生式继承+构造函数继承

之前我们一直没有解决的问题是，继承原型链上的引用数据类型是都只是保留了一份引用，这意味着所有子类都能共用一份数据

这意味着巨大的污染风险，而寄生组合式继承就能帮助我们解决这种问题

寄生式继承的核心是把父类作为子类的原型，

寄生组合继承的核心是在这个基础上，把父类的原型深拷贝一份，再处理父类实例上的数据

这里先介绍一个方法：

 Object.create = function(obj) {
      function F () {}
      F.prototype = obj
      return new F()
 }

let obj = {
    color: "red"
}

let a = Object.create(obj)
let b = Object.create(obj)

a.color = 'green'
console.log(a.color, b.color) //green red

可以看到这样就能简单地实现深拷贝

为什么能够实现深拷贝？F.prototype只是保留了对obj的引用，那new了之后应该还是引用才对，但是为什么ab没有污染？

因为参数是有形参和实参之分的，形参就已经创建了实参的副本？

这一点，我们放到后续章节 JavaScript的求值策略，

function inherit (child, parent) {
    child.prototype =  Object.create(parent.prototype) 
	// 这一步的意义在于规范化，无实际影响
    child.prototype.constructor = child 
}

function Planet (name) {
    this.name = name
}
Planet.prototype.roll = function () {
    console.log("天地玄黄，宇宙洪荒")
}

function Earth (name, size) {
    Planet.call(this, name)
    this.size = size
}
inherit(Earth, Planet)
Earth.prototype.creatCreature = function () {
    console.log("Hello, here is" + this.name)
}
const earth = new Earth('地球')
earth.createCreature()
earth.roll()

它是目前提到的继承方式中最完美的一种

具有原型链上数据不污染，子类可扩展等等有点

求值策略

在之前讨论寄生组合式继承时，遇到了关于如何JS的函数参数传递究竟是如何进行问题——引用传递，值传递 或者说 其他可能？

引用传递

让我们先来进行一些实验：

let a = 1, b = 2

function exchange (p1, p2) {
    let temp = p1
    p1 = p2
    p2 = temp   
    console.log("函数中") 
    console.log(p1, p2) // 2 1
}

exchange(a, b)
console.log("全局")
console.log(a, b) // 1 2

这看起来是引用传递对吧？

我们再来对引用数据类型试试（节约篇幅，不再复写exchange函数）：

let c = {name: '张三'}
let d = {name: '李四'}
exchange(c, d) // 内部输出 {name: '李四'} {name: '张三'}
console.log("全局")
console.log(c, d) // {name: '张三'} {name: '李四'}

ObjMaker(c)
console.log(c.nickName) // 王五

等等，如果是引用传递，那么就会有形参和实参之分，也就意味着这里的nickName只是形参的属性 而 实参是不应该具有的——那么这里的情况看上去又是 值传递

共享传递

也叫 对象传递，可以看做是 引用数据类型 + 引用传递， 规则是在引用数据类型（对象）时，参数是其引用的副本（不是其引用，也不是其副本，是其引用的副本！）

仔细想一下非常好理解，因为let a = {} 中a就是引用，其副本的意思就是b = a，这就意味着改变a或者b 的指向不会对原对象产生任何影响，但是如果试图修改原对象身上的属性，那么由于它们都指向了原对象，所以访问到的是同一份数据

这就解释了我们所看到的一系列情况

总结

最后总结一下就是

JavaScript中函数的求值策略是，对于基本数据类型采用引用传递，对于引用数据类型采用共享传递

暂时性死区

即TDZ （Template Dead Zone）

我们先看一个例子：

if (true) {
    a = 10
    console.log(a)
}

都看到这里了，我相信大家一定都知道运行结果是10对吧，这里比较特殊的就是a没用变量声明的关键字，会被直接挂载到window上（其实这样写得话，在ES5没有块级作用域，所以web环境中相当于直接写到了window上）

如果这个没有问题，那么下面这个又如何呢？

if (true) {
    a = 10
    console.log(a)
    let a
}

先想想，如果不是let而是var，那么由于变量声明提升的缘故，结果是不会发生变化的

但是let的存在使得这里的ES6块级作用域变得诡异，在这个作用域内，let a上面的区域，就是变量a的暂时性死区，变量a在它的暂时性死区里面，是不能够被访问到的，所以结果会 报错

原理是，预解析阶段，JS检查到这个let a时，便认为在这之前a变量是不存在的，所以在此前进行访问a会报错

再来看一个：

if (true) {
    var a = 10
    console.log(a)
    let a
}

依然是报错，但是报错的类型不同，这里报错是因为**let不允许重复声明a**
