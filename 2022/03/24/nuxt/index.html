开始使用Nuxt

概念

CSR和SSR

CSR即客户端渲染，Client Side Render

SSR即服务端渲染, Server Side Render

CSR

一开始没有数据，是通过前端模板（如Vue React等）引擎注入到页面内部，
不利于SEO，爬虫来的时候还没有太多数据

SSR

利用后端的渲染引擎（如Next， Nuxt），把我们提供的数据注入，然后完成渲染后将完整的页面结构返回，
数据无需再渲染，因为数据已经渲染完毕了
利于SEO，但是对服务器要求较高

一般开发是CSR和SSR混合使用

Vue的SPA结构的缺陷

SPA结构优点是只需要做一次请求，缺点则是SEO不友好，除非渲染能够在服务端完成，这就是Node的真正用处——SSR服务端渲染

渲染机制的探索历程

最早的JSP和PHP就是纯SSR，后来有了CSR，现在又到了CSR+SSR

简单使用

基本渲染

const Vue = require('vue')
// 创建一个Vue的实例
// 注意这是Vue2.x才能new
const app = new Vue({
    template: `<div>HelloWorld</div>`
})

const renderer = require('vue-server-renderer').createRenderer()

// 将Vue示例渲染成HTML
// 渲染好了的html会被放入回调中
renderer.renderToString(app, (err, html) => {
    if(err) {
        throw err
    }
    console.log(html)
    // <div data-server-rendered="true">HelloWorld</div>
})

Promise化渲染

// renderToString在该包的2.5+版本中还支持promise
renderer.renderToString(app).then((html) => {
    console.log(html) 
    // <div data-server-rendered="true">HelloWorld</div>
}).catch((err) => {
    console.log(err)
})

响应到浏览器展示

这里我们使用express框架来做这个事情

const Vue = require ('vue')
const server = require ('express')() // 注意express返回的是函数

// 访问根目录时，创建一个Vue实例
server.get('/', (req, res) => {
    const app = new Vue({
        template: `<div>HelloWorld</div>`
    })    

    const renderer = require('vue-server-renderer').createRenderer()

    renderer.renderToString(app).then((html) => {
        // console.log(html)  
        res.send(html)
    }).catch((err) => {
        console.log(err)
    })
})

// 接下来访问localhost就好了
server.listen(3001, () => {
    console.log('服务器在3000端口上打开')
})

这里就要提一下，res常用的方法一共有四种，分别是：

res.set(), 可以设置响应头等信息

res.json() res.end() res.send()
三种方法都能响应给前端，但是它们有所区别

认识Nuxt生命周期

在Render之后，才会开始Vue的生命周期，

Vue的两个钩子——创建前和创建后是服务器端到浏览器端的过渡，所以它们两个是特殊的生命周期钩子

另外，active这类可以重复的钩子不包括在内

我们解析一下各个生命周期:

  1. nuxtServerInt 其中有些钩子只会执行一次，一般用于执行初始化数据
  2. 可以在图中的三个位置进行配置，在某个页面、组件加载前运行
  3. 实现成功则进入页面，否则404这样的功能
  4. 组件初始化前调用，要注意此时的this指向
  5. 开始渲染，此时如果请求新的页面，那就跳回第2步

CSR生命周期

这里开始就是Vue自带的一些钩子了

// 从这里开始就是CSR了

beforeMount () {},

mounted () {},

beforeUpdate () {},

updated () {},

beforeDestroy () {},

destroyed () {}

// 注意服务端不存在 activated和deactivated，因为服务端无法缓存

我们提到过beforeMount和mounted比较特殊，是灰色地带，它们虽然是CSR生命周期，但是它们也不能保证能够成功访问到window等客户端的内容

它们的运行机制是，先试图在客户端运行，如果此时客户端来不及加载，那么则在服务端运行

Nuxt路由

约定式&&声明式

声明式
需要做文件的导入和声明等工作才能用，虽然麻烦但是灵活

约定式
不需要做文件的导入等工作，只需要按照一定的规定做就可以使用，Nuxt路由可以使用约定式，只要建立在pages下的文件，就能够直接访问到

其中goods的二级路由里面有一点需要注意，这里命名为_id.vue，_表示这个id是一个动态的，这个二级路由是动态路由

看到这里我们也就明白了约定式的多级路由就是通过文件层级来判断的级别，通过文件名称来判断从属关系

注意这里二级路由还没有展示区，我们需要给一个<nuxt />给予展示
