前置概念

强弱类型

这是从类型安全的角度区分的

强、弱类型缺乏官方的界定标准，但是较为认可的一种说法是：

强类型语言中不允许任意的隐式类型转换；

弱类型语言则允许。

静动态类型

这是从类型检查的角度区分的

有较为官方统一的解释：

静态类型即，变量声明后就不允许再修改了；

动态类型则允许，并且在运行的时候才能明确一个变量的类型

注意，并非弱类型就是静态类型，强类型就是动态类型

类型

概念

TypeScript是JavaScript的超集，

缺点一：增加了如接口，泛型，枚举等的“新概念”（？迷惑，却不说）
但是好消息是，TypeScript是渐进式的
也就意味着，就算我们什么新概念都不知道，凭借JS基础我们依旧能够直接开始使用它
缺点二：项目初期，TypeScript会增加开发成本
大概是需要自定义各种数据类型等等工作加大了开始的成本，不过这便于后期的维护

类型推断

隐式类型推断

即在不明确变量类型的时候，系统根据其使用情况来判断它是什么类型

比如

let a = 123 // a的类型被推断为number
a = 'hhh' // 报错，因为'hhh'不是number

显示类型声明

相较于隐式类型推断，显示类型声明采用了类型注解的方式确定了变量类型

function sum(a: number, b: number): number {
	return a + b
}
const arr: Array<number> = []; // 数字数组

控制流分析 和 类型收窄

// 这叫做 联合类型
function doSomething(data: string | number | boolean) {
    if (typeof data === 'string') {
        return data + '2333';
    } else if (typeof data === boolean) {
        return !data; 
    } else {
        return data * 2;
    }
}

上面return的三处均为发生类型报错，这是因为TS通过类型的控制流分析进行了类型的收窄

可以看做是，TS由上往下地检查代码，根据if-else等条件语句不断地剔除data当前多余的类型

比如这里的else中，data的类型已经被剔除为 number了，所以就不会报错——这种类型范围不断收缩明确的过程就是 类型收窄

类型断言

开发者的担保

const num1 = res as number; // 写法一
const num2 == <number>res; // 写法二

这样写就是说，开发者担保res是number类型的

TS虽然能够推导类型，但是肯定不如开发者那样了解代码的逻辑，所以TS有时会对当前的类型表示怀疑而报错，这时候我们用类型断言就能消去这种报错

比如这样：

function add(maybeNum): number {
    return (maybeNum as number) + 1;
}

这里我们我们没有限制maybeNum的类型，但是我们担保它是一个number，所以这里进行运算也不会报错

另外，如果你不确定某个变量的类型，并且是实在没有别的办法了，那么试试any类型

const str = '';
console.log((str as any).name); // 不报错

any，包含了所有TS中能够出现的类型，可以看做凌驾于所有类型之上，无视TS类型规则

极其不符合

虽然开发者担保了，但是TS也不会完全就此罢休

let strNum = '123';
function add(): number {
    return (strNum as number) + 1; // 报错
}

很显然，这里strNum已经明显不是一个number了，这里担保显得非常荒诞，所以TS依然会报错

如果要取消这种报错，我们可以这样做：

let strNum = '123';
function add(): number {
    return (strNum as unknown as number) + 1;
  // return (strNum as any as number) + 1; // 同理
  // return (strNum as any) + 1; // 亦可
}

这里有一个新类型unknown，它的含义是 目前是未知的，但是一定会是某一个类型

比如一个网络请求，在真正的数据响应回来之前，无论其值如何声明，都可以看做是unknown

因为TS是静态类型分析，本身不会在运行时做校验

TS基础数据类型

既然有了类型，那么究竟有哪些类型呢？

原始类型

就是我们熟知的JS三大基础类型

let a:number = 1024
let b:string = 'hello world'
function c ():boolean {
    return false
}
// 当然，函数也不一定要加类型注解，因为隐式类型推断对函数仍然起作用

Array类型

类型注解等不能直接写array，而是要像下面这样:

let arr1:number[] = [1, 2, 3]
// arr2这里采用了一种泛型的写法，后面会细说
let arr2:Array<number> = [1, 2, 3]

any类型

设置为any之后，那么就可以像原来JS那样自由灵活了

这里还有一个需要补充的点就是关键字noImplicitAny，可以在tsconfig.json文件中将这个字段设置为true，此后当变量被隐式类型推断为any的时候将出现报错

Object类型

说实话，感觉这样挺奇怪的

let obj:{x: number, y: number, z?: number} = {x: 123, y: 456}
// 对于对象的属性的类型，是可选的，不写默认为any类型

加上问号代表这是一个可选的属性，非必填项

但是这样就意味着，你在使用它之前，需要校验是否为undefined

TS高级数据类型

联合类型

类型收窄，一种方式是通过使用联合类型实现（另一种是通过if语句结合typeof关键字来做判断，即 类型保护方式实现）

let a = 123 | 456
// a = 789 // 报错了，不是123和456中的一个

let b = number | string | string[] 
b = 123
b = '456'

观察上述代码，可以知道，可以写多个类型也可以具体到值

同样地，在使用它们之前，可能需要做类型校验

自定义类型

通过给出 类型别名 的方式来自定义类型， 关键字是type

type Position = {
    x: number,
    y: number,
    z?: number
}
let p:Position = {
    x: 123,
    y: 456
}

可以用type给任何类型自定义一个名称——当然包括联合类型

没错，这种操作非常符合我们的逻辑思维

Interface类型

interface Position{
    x: number,
    y: number
}
let p:Position = {x: 123, y: 456}

诶，看着怎么和type自定义类型一样？他们的区别是什么呢？

  1. interface可以重复声明，同名自动合并
  2. interface声明的内容必须是抽象的
  3. 可以用implements实现接口

declare类型声明

一开始我看到类型声明的时候还愣了一下
为了避免误解我还是在其前面加上一个declare吧

通常是用于处理外部引入的、没有指定参数、返回值类型的函数，对其追加参数、返回值的类型声明

import { camelCase } from 'loadsh'

// 模块中提供的这个函数并没有定义参数和返回值类型
// 我们用以下形式给它补上这些信息
declare function camelCase (input: string): string

const res = camelCase('hellotankyoutankyouverymuch,iamok')

接口

基本使用

其实吧，我寻思啊，这js怎么还整接口啊…

// 这是接口
interface Post {
	title: string
	content: string
}

// 写个函数
function print(p: Post){
	console.log(p.title, p.content)
}

// 调用函数
print({
	title: 'Hello, Interface'
	content: 'TypeScript is so cool'
})

成员修饰

可选成员
即可有可无的成员

// subtitle后面加上了一个? 表示这个成员可有可无
// 原理是标记subtitle类型是string 或者 undefined
interface Post {
	title: string
	subtitle?: string
	content: string
}

只读成员
不可手动设置和修改

interface Post {
// readonly关键字约束成员即可设置只读
	title: string
	subtitle?: string
	content: string
	readonly summary: string
}

动态成员
有些时候一个接口该有哪些成员是不确定的

// [prop: str]设置成员名称为str
// 其后的string指定值为string
interface Cache {
	[prop: str]: string
}

//使用
const cache: Cache = {}

cache.foo = 'value1'
cache.bar = 'value2'

类

对ES6引入的类进行了扩展

基本使用

class Person {
	name: string // = '赵宇'
	private age: number
	constructor (name: string, age: number) {
		this.name = name
		this.age = age
	}
}


const Y1M = new Person('杨二鸣', 20)

注意，在TS标准中，类的成员变量要么初始化，要么用构造函数设置值
另外，如果不加修饰，那么默认为public，不是default
当然还有，protected，和private区别大概是

protected可以继承，只能在子类中访问（子类的实例对象不可访问）

private不可继承，只能在当前类中访问

同样，继承extends，静态static，只读readonly， 抽象abstract这些东西也被加入到了这个体系，
总之就是Java玩剩下的东西都被拉到了TS的体系中

这里单独说一下readonly，
放到public等修饰的后面，
被其修饰的成员先得在声明时或构造函数中初始化，此后不能再被修改

类和接口

Java的类和类之间是单继承的，类和接口之间是多继承的，
所以TS这里类和接口的关系还是多继承的（蜜汁逻辑）

用法和Java一样，依旧是用implements实现
书写的规范是，一个接口约束一个有关功能即可

Getter和Setter

除了像Java那种广为人知的方式设置Getter和Setter，TS还为我们提供了一些更为简洁的语法

class SomeClass {
	private name: string
	constructor (name) {
        this.name = name
    }
    
    /* 这是经典的方式
        getName () {
            return this.name
        }
        setName (name) {
            this.name = name
        }
    */
    get name () {
        return this.name
    }
    set name (name) {
        this.name = name
    }
}

const someObj: SomeClass = new SomeClass("张三")
someObj.name = '李四' // 自动调用set

泛型

果然，我就知道泛型也跑不掉

基本使用

function createArray<T> (length: number, value : T): T[] {
	const arr = Array<T>(length).fill(T)
	return arr
}

const res = createArray<string>(3, 'foo')
