React渲染的过程

  1. 利用babel将jsx转化为js
  2. 利用React对象的createElement方法来创建虚拟DOM，需要三个参数：
    type(标签类型)、props(属性)、children(子节点)
  3. 转换为真实DOM
  4. 利用render方法渲染

让我们来观察一下这个过程对应的代码（react18，工程项目中的index.js）：

import React from 'react';
import ReactDOM from 'react-dom/client';
import './index.css';
import App from './App';
import reportWebVitals from './reportWebVitals';

// 这种声明形式实际上是React.createElement的一个语法糖，这样更直观简洁
const ele = (	
    <React.StrictMode>
        <App />
    </React.StrictMode>
)
// 获取要挂载到的根节点
const root = ReactDOM.createRoot(document.getElementById('root'));
// 利用render函数渲染到目标上
root.render(ele);

  1. class需要写作className

  3. 需要有唯一一个最外层元素包裹

状态state

之前说了函数组件式无状态的，相对地，类组件就是有状态的（state），其实就是类似于Vue中的data这么一个概念

我们常用以下两种形式来修改state:

this.setState({ count: this.state.count + 1 });
this.setState(prevState => ({ count: prevState.count + 1 }));

第二种写法适用于一些我们需要获取上个state的信息的场景

这里有个细节，如果每次调用setState就会渲染一次，那么高频调用会导致开销极大

所以react和vue对这种情况的处理是先记录这个变化，等到一系列更新操作结束再统一处理

直接修改是没有效果的，因为它修改的是虚拟DOM，而react没有数据劫持所以没有办法直接更新到真实DOM

生命周期

这里分两个部分介绍，分为React18以前和React18

旧版本

React18以前的旧版本生命周期相对精简一点，方便入门理解:

语义化非常强，所以理解起来不算困难

需要注意，constructor也算是一个生命周期

React18

更新的内容简单总结一下就是：

  1. 移除了3个带will的生命周期钩子（仅剩下commponentWillUnmount）

    如果要继续使用这些被移除的钩子，那么需要加上前缀UNSAFE_

    但是正如其名，它们是随时可能被彻底移除的，建议避免使用

  2. 更新了2个新生命周期钩子

    1. getDerivedStateFromProps

      这个函数前面应加static，因为是静态的
      通过这个钩子修改的state，state任何时候都取决于props，其他的函数无法进行修改

    2. getSnapshotBeforeUpdate

      在更新之前获取DOM信息

this指针

问题情景

想一下这个情景，如果要有一个点击事件，那么我们就需要在类内部为之声明一个方法作为回调

注意，这里的类没有被实例化，方法内部this还指向undefined（严格模式）

所以这里this没有被指向调用者，当我们试图访问this，会发现它是undefined（类中默认开启严格模式，否则指向window）

解决方案

可以在构造器里面，用bind将this指针绑定，也可以利用箭头函数的特性

注意call和bind、apply的区别，call、apply会直接执行，而bind会返回一个新的函数而不会执行

还有就是不要忘了class是封装的function，所以其this指向还是可以像function的那样分析

class Cla extends React.Component {
    constructor () {
        super ()
        this.state = {
            
        }
        this.oneMethod = this.oneMethod.bind(this)
    }
    oneMethod() {
    	console.log(this) //     
    },
    anotherMethod = () => {
        console.log(this)
    }
    
    render () {
        return (
        	<button onClick={this.oneMethod}>点我!</button>
        	<button onClick={this.anotherMethod}>点我!</button>
        )
    }
}

父传子
root.render(<Component2 someProps="hhh">123</Component2>)

这里可以传参对吧，那么父级在调用子级组件的时候，就可以给自己帮一个参数

属性部分参数会被props属性接收，通过this.props.someProps就可以访问到上述传参(但是它是只读的)

标签所包裹的部分，会被props.children属性接收, 通过this.props.children就可以访问

这里我们额外补充一些关于参数传递的东西

批量传递这里涉及到一点容易混淆的东西

let obj = {
    name:'王尔德'
}
root.render(<Component2 {...obj}></Component2>)

这里使用了展开运算符，这里做三点解释：

  1. 展开运算符是不能够直接直接展开对象的，需要在两边加上花括号才能使用，并且这样做是深拷贝
  2. 扩展运算对象的同时还可以继续追加字段，可以作为添加或者合并对象的方式，字段重复则以后者为准

在知道了以上两点之后，React的特性——

  1. JSX中的{}表示使用的是JS，所以{…obj}的意思是在标签中直接展开的，然后另一边的props也能直接收到
    这并非是对象展开，而是JSX的特性

另外，如果要对参数做限制，那么需要写一个propTypes字段，其中写下各个参数的限制：

class Hhhh extends React.Component {
    constructor() {}
    render() {}
}

Hhhh.propTypes = {                
    // 限制类型为string, 且必传
    // 这个写法需要引入一个包叫做prop-types
    name: PropTypes.string.isRequired
    // 限制类型为函数
    speak: PropTypes.func
    // react-16及以下版本，需要写为Person.PropTypes.string
}

// 当然, 也可以设置默认值
Hhhh.defaultProps = {
    age: 18
}

注意，我们这里是怎么写的？是类名.字段，是没有进行实例化就直接调用的——相当于是添加静态属性:

class Hhhh extends React.Component {
    constructor(props) {
        // 得调用父级的构造函数，防止使用时未定义变量的bug
        super(props)
    }
    render() {}
    static propTypes = {                
        name: PropTypes.string.isRequired
        speak: PropTypes.func
    }
    static defaultProps = {
        age: 18
    }
}

其实现在PropType用的不多，更多是用的TS做类型限制

子传父

这里是react比vue方便的一个地方，react中可以直接将父级的一个方法传递给子级，子级直接调用这个方法能够操作到父级的一些数据（有点闭包那个意思）

vue中还得指定自定义事件，通过emit来触发…

同级互传
refs

这里有三种写法，时间顺序由上到下

第一种:字符串形式
可以说是轻车熟路了，通过ref绑定，this.refs访问，存在效率上的问题，未来的版本中可能被移除

第二种:回调形式
绑定的时候写成ref={(a) => {this.xxx = a}}，因为这段JSX在render阶段会被自动调用，并且第一个参数自动传当前节点

这种写法存在一个问题就是，如果以内联函数的形式定义（其实就是说在里面直接写函数表达式），那么在更新的过程中（更新是指通过setState等操作更改，不包括初始化和输入等操作)会执行两次，并且第一次传参为null，第二次传参才是DOM元素

为了避免上面说的情况，需要将函数剥离出来，把函数的引用赋值给ref就可以了

第三种
绑定的时候写成ref={this.some}，然后访问时先要写some = React.createRef()，这样之后可以得到一个容器，这个容器可以存储所有被ref标识为some的节点

双向数据绑定

原理就是通过onChange事件来监听数据变化，然后在事件处理函数中更改state就可以实现双向绑定了

注意，如果输入框绑定了状态值，因为绑定不是双向的，输入新内容的时候状态值不会自动更新，
然后反过来，又因为绑定的缘故，输入的新内容也会失效——一个微妙的死锁就产生了：

函数柯里化的运用

平时确实没怎么用过，这里终于有个用处了。

需求是：

实现一个用户账号和密码的双向绑定，但是onChange只能绑定一个函数

按照正常的思路来说，我们会为onChange写一个函数，然后传参是dataType和event分别代表数据类型和事件对象，便于区分账号和密码以及获取对应的值

但是问题就在这里了，下面这两种写法会报错

 <input onChange={this.saveFormData('username', {e})} value={this.state.username} name="username"></input>
 
<input onChange={this.saveFormData('username', e)} value={this.state.username} name="username"></input>
{/* e is not defined */}

传参固然有很多种写法，这里演示一种柯里化的写法

saveFormData = (dataType) => {
        return (event) => {
            this.setState({[dataType]:event.target.value})
            // 注意这里的[dataType]加了方括号
            // 类似于this.state[dataType]
            // 不加就是this.state.dataType
        }
    }
    
    render () {
        return (
            <React.Fragment>
				<input 
                    onChange={this.saveFormData('username')} 
                    value={this.state.username} 
                    name="username"></input>
				<input 
                    onChange={this.saveFormData('password')} 
                    value={this.state.password} 
                    name="password"></input>
            </React.Fragment>
        )
    }

这里传参很有讲究，首先我们绑定事件的时候，是写的函数的引用——也就是不加括号（加上括号在解析的时候就会自动调用了，不会获得事件对象），这里取的是返回的结果——一个函数的引用，这下就绑定上了onChange事件，于是乎就能够获得事件对象event，非常巧妙

按照这个逻辑，写成onChange = { (event) => {} }也可以

此外，其实可以不传dataType，这个可以通过事件对象获取

受控和非受控组件

非受控组件

受控和非受控组件主要是针对于表单的，

非受控组件简单描述就是

HTML表单有自己的状态（state），能随着用户的输入，在UI上更新——这种更新就是不受控制的

这种表单组件就是非受控组件

受控组件

就是利用onChange进行控制

利用onChange让React的状态和表单的状态相互关联，从而使React能够使用setState等方法对表单进行控制

这种表单组件就是受控组件

介绍一些和Vue不太一样的内容

CSS的模块化

Vue中有scoped可以用，但是React没有，这导致最后所有CSS到了App.jsx中就会发生冲突

解决方式就是，将CSS文件命名为形如xxx.module.css的形式，然后像下面这样使用：

import React, {Component} from 'react'
// import './xxx.css' 这是不采用模块化的形式
import myCSS from './xxx.module.xxx'

export default class MyComponent extends Component {
    render(){
        return <div className={myCSS.someClass}>Hi!!!</div>
    }
}

代理配置

简单配置

这里可以在package.json中新加一个proxy字段，值直接写后端的地址即可——当然这是最简单的一种写法，稍微详细一点的可以像以前配置Vue的Proxy那样写

只配置一个简单的代理的话，直接在package.json中写：

proxy: 'http://localhost:5500'
详细配置

如果要写更复杂乃至多个代理，那么需要在src下建立一个setupProxy.js文件（这里是约定大于配置，所以名字不能改）

// 没错，是CommonJS,这里用中间件建立了一个代理服务器
const proxy = require('http-proxy-middleware')
module.exports = function (app) {
    app.use(
    	proxy('/api1',{
            target: 'http://localhost:5000',
            changeOrigin: true, // 意向的是请求报文的Host报文
            pathReWrite{'^/api1': ''}
        }),
        proxy('/api2',{
            target: 'http://localhost:5001',
            changeOrigin: true, // true是显示target的值，false就是显示真正前端的值
            pathReWrite{'^/api2': ''}
        })
    )
}

它们实际上都是一个东西，即webpack的devServer

当前端需要但自身没有某个资源的时候，就会去找代理进行请求

这里就来复习一下跨域的几种解决方式:

JSONP
img、script标签通过src属性获取资源不受跨域限制，可以利用这点获取数据

CORS

性能优化

我们将通过Component、PureComponent、memo三者来进行性能的优化

在React中，代码重用的方式主要是组合而不是继承，即使是facebook的代码中，也没有出现继承（除了继承Component等基类以外），而是通过import引入再用组件包含的形式来进行开发

为了节省更新UI的开销，我们可以通过shouldComponentUpdate返回的布尔值来判断是否更新组件

像这样：

Component

class CounterButton extends React.Component {
  constructor(props) {
    super(props);
    this.state = {count: 1};
  }
  // 关键是下面这段
  shouldComponentUpdate(nextProps, nextState) {
    if (this.props.color !== nextProps.color) {
      return true;
    }
    if (this.state.count !== nextState.count) {
      return true;
    }
    return false;
  }

  render() {
    return (
      <button
        color={this.props.color}
        onClick={() => this.setState(state => ({count: state.count + 1}))}>
        Count: {this.state.count}
      </button>
    );
  }
}

Hook

本来该放到顶层API目录下，但是这块儿相当重要，就连官方文档都是另起章节

概念

一个新概念的产生必然是为了解决或者至少改善既有的问题：

  1. 组件间复用状态逻辑很困难，现有的解决方式也容易产生”嵌套地狱”等问题
  2. 生命周期内的代码关联性混乱（类比Vue的选项式API）
  3. class的this、使用场景等问题

然后就有了React16.8的新特性——Hook

Hook 是一些在 函数组件 里“钩入” state 及 生命周期等特性 的 函数

它们的特征是，名称都以use开头

注意Hook不能在Class类组件中使用

当然，React做了很好的向后兼容，没有打算移除Class，而且Hook能够和现有的代码同时工作！

State Hook

直接看代码

import React, { useState } from 'react';

function Example() {
  // 声明一个叫 “count” 的 state 变量。
  const [count, setCount] = useState(0);
  // 多个声明
  // 可以放到一个对象里————这取决于是否利于业务逻辑
  const [fruit, setFruit] = useState('芭娜娜');
  const [todos, setTodos] = useState([{ text: '摸鱼划水' }]);
  
  return (
    <div>
      <p>点了 {count} 次</p>
      <button onClick={() => setCount(count + 1)}>
        点我啊！
      </button>
    </div>
  );
}

这里调用的useState就是一个hook，它返回一对值，分别是state和更新这个state的函数, React每次渲染后会保留这个state

useState的参数将作为state的初值，这个参数仅在组件第一次被渲染时被使用

这就是一个问题的答案：为什么是叫做use而不是create？
这是因为create不准确——因为只有首次渲染该组件的时候才创建了state

Effect Hook

可以把 useEffect Hook 看做 componentDidMount，componentDidUpdate 和 componentWillUnmount 这三个函数的组合

import React, { useState, useEffect } from 'react';

function Example() {
  const [count, setCount] = useState(0);
  // 这里的效果与 componentDidMount 和 componentDidUpdate 生命周期类似
  useEffect(() => {
    // 更新标签页的标题
    document.title = `点了 ${count} 次`;
  });

  return (
    <div>
      <p>点了 {count} 次</p>
      <button onClick={() => setCount(count + 1)}>
        点我啊！
      </button>
    </div>
  );
}

代码应该很容易读懂，但是effect hook到底做了什么？

通过传递回调函数（即effect），告诉 React 组件需要在渲染后执行的操作，React 会保存effect，并且在执行 DOM 更新之后调用

这里传递回调函数是箭头函数形式的，这使得每次渲染后都会创建一个全新的回调函数，保证了其内部的值是最新的

另外，effect可以返回一个函数，在componentWillUnmount的时候自动执行——这就对应了开头那句话，它可以看做三个生命周期的组合

如果要使用多个Effect Hook

function Form() {
	
  const [name, setName] = useState('张三');
  useEffect(function persistForm() {
    localStorage.setItem('formData', name);
  });

  const [surname, setSurname] = useState('李四');
  useEffect(function updateTitle() {
    document.title = name + ' ' + surname;
  });

}

Hook使用规范

核心就是只在最顶层使用 Hook，不要在循环、条件语句、函数等次级作用域中使用，如果必须要有条件语句等，那么可以考虑将其放到effect内部

useEffect(function persistForm() {
   // 👍 将条件判断放置在 effect 中
   if (name !== '') {
     localStorage.setItem('formData', name);
   }
 });

为什么不能放到条件判断等语句内？

因为Hook的调用顺序必须始终一致，不能允许”这次调了，下次不调“这种情况，比如这里有4个Hook，分别叫做A,B,C,D

第一次调用
A->B->C->D
第二次调用
A->C->D // 出错

具体一点，可以是像这样：

// ------------
// 首次渲染
// ------------
useState('Mary')           // 1. 使用 'Mary' 初始化变量名为 name 的 state
useEffect(persistForm)     // 2. 添加 effect 以保存 form 操作
useState('Poppins')        // 3. 使用 'Poppins' 初始化变量名为 surname 的 state
useEffect(updateTitle)     // 4. 添加 effect 以更新标题

// -------------
// 第二次渲染
// -------------
useState('Mary')           // 1. 读取变量名为 name 的 state（参数被忽略）
useEffect(persistForm)     // 2. 替换保存 form 的 effect
useState('Poppins')        // 3. 读取变量名为 surname 的 state（参数被忽略）
useEffect(updateTitle)     // 4. 替换更新标题的 effect

// -------------
// 第三次渲染
// -------------
useState('Mary')           // 1. 读取变量名为 name 的 state（参数被忽略）
// useEffect(persistForm)  // 🔴 此 Hook 被忽略！
useState('Poppins')        // 🔴 2 （之前为 3）。读取变量名为 surname 的 state 失败
useEffect(updateTitle)     // 🔴 3 （之前为 4）。替换更新标题的 effect 失败

当然我们目前只是简单说明了这个BUG产生的条件和现象，并没有从底层源码上剖析成因——不过应该会有这个机会的

自定义Hook

由于Hook本身是函数，函数内又可以调用其他函数，所以我们可以基于现有的Hook来自定义Hook

import { useState, useEffect } from 'react';

function useFriendStatus(friendID) {
  const [isOnline, setIsOnline] = useState(null);

  useEffect(() => {
    function handleStatusChange(status) {
      	// ...
    }
	
    return () => {
    	// ... 
    };
  });
 // 注意Hook的定义，没说一定要返回一对值
  return isOnline;
}

Hook不一定只是为了处理数据，更可以把它当成一种mixin，实现逻辑的复用——这也遵循编程的基本思想，将重复的逻辑封装成一个函数方便复用

而且这也利于Hook之间的通信——一个Hook可能需要一些参数才能得到想要的结果，而这些参数又是需要另一个Hook产生，自定义Hook使得它们之间的参数传递更加灵活

比如这样：

const [recipientID, setRecipientID] = useState(1);
const isRecipientOnline = useFriendStatus(recipientID);

  }
}

// 声明Hook，之后调用就把todosReducer传给reducer参数
function useReducer(reducer, initialState) {
  const [state, setState] = useState(initialState);

  function dispatch(action) {  
    const nextState = reducer(state, action);
    setState(nextState);
  }

  return [state, dispatch];
}

// 最后在组件中调用这个Hook
function Todos() {
  const [todos, dispatch] = useReducer(todosReducer, []);

  function handleAddClick(text) {
    dispatch({ type: 'add', text });
  }

  // ...
}

diff算法

通用的情况下是一个O(n^3)的算法，n是虚拟DOM的个数，这个代价实在太大，所以React在以下两个假设的基础上提出了一套O(n)的算法:

  1. 两个不同类型的元素会产生出不同的树
  2. 开发者可以通过 key prop 来暗示哪些子元素在不同的渲染下能保持稳定

据说在实践中，几乎所有场景下上述两个假设都成立

key的作用

当状态发生变化的时候，React会根据新的数据生成新的虚拟DOM，然后新的虚拟DOM与旧的虚拟DOM进行比较，规则如下：

如果存在相同的key，那么检查内容，如果内容没变则直接使用之前的真实DOM；否则生成新的真实DOM替换；

如果不存在相同的key，那么直接生成新的真实DOM进行渲染。

过程分析

先放一张图具体一点

这里可以看出，如果使用下标作为key值，那么逆序插入、非末尾删除等破坏原有的绝对顺序的操作会引起效率乃至效果上的问题

更具体的比较过程是这样，分几种情况进行讨论：

1.比对不同类型的元素

<div>
  <Counter />
</div>
<!-- 更新前后 -->
<span>
  <Counter />
</span>

根节点不同时，整棵树都会被拆卸，所以这里<Counter/>也会被拆卸然后重新挂载

2.比对同一类型的元素

<div style={{color: 'red', fontWeight: 'bold'}} />
<!-- 更新前后 -->
<div style={{color: 'green', fontWeight: 'bold'}} />

甚至不会更改整个style，仅更改color属性的值

在上述步骤完成后（无论是情况1或情况2），React调用componentWillReceiveProps和componentWillUpdate，然后在子组件中递归diff——亦如上面那样，React尽可能只更新需要更新的部分

比如：


const root = ReactDOM.createRoot(
  document.getElementById('root')
);

function tick() {
  const element = (
    <div>
      <h1>Hello, world!</h1>
      <h2>It is {new Date().toLocaleTimeString()}.</h2>
    </div>
  );
  // 最后只更新花括号里面的内容，甚至It is都不会被更新
  // 虽然每次render都会新建一个描述整个UI树的元素，
  // 但是这里React实际更新的只有变化的文本节点
  root.render(element);
}

setInterval(tick, 1000);
